Evolutionary multi-agent framework

Idea Evolve

Autonomous framework where specialized Claude agents work in parallel across generations and build a shared knowledge base that gets smarter every generation — no human in the loop.

Problem

Why single-agent LLM prompting stalls on hard problems

Hard optimization problems resist single-shot LLM prompting. A single agent gets stuck in a local optimum, repeats the same ideas, and stops improving. The solution space is too large for one perspective to cover.

Best-of-N sampling from one model helps a little, but the samples are drawn from the same distribution — you get variance, not diversity. The failure mode is recognisable: after a few tries the model cycles through the same three approaches, never finding the fourth.

Approach

Role-specialized agents plus a shared, growing knowledge base

Idea Evolve runs 3–8 specialized agents in parallel inside each generation. Each agent writes candidate solutions and iterates on them for 40+ cycles. The best survive, get combined, and seed the next generation — a classic evolutionary loop, except the variation is produced by agents with different roles instead of random mutation.

What makes it compound is a shared knowledge base: after every generation an Evaluator extracts ideas, patterns, and facts into a growing hierarchy (State of Affairs → Clusters → Ideas/Patterns/Facts). The next generation's agents read that knowledge before writing anything, so lessons transfer across runs — the system literally gets smarter each generation.

Pipeline

Six phases per generation

A stateless orchestrator runs every generation through the same six phases. All state lives in files — if the orchestrator crashes, restart the same command and it resumes exactly where it left off, skipping completed agents and killing orphans.

Architect
Plans the generation's strategy and assigns specialized agents.
Agent work
3–8 agents work in parallel — writing solutions, evaluating them, iterating 40+ cycles each.
Evaluator
Scores every candidate and extracts knowledge (ideas, patterns, facts) into the shared base.
System critic
Diagnoses pipeline issues and recommends improvements for the next generation.
Consistency
Audits the knowledge base so contradictions and duplicates don't compound.
Finalize
Updates rankings, scores, and detects any manual interventions before handing off.

Pipeline tab — per-agent status (waiting / running / done / failed) and the data flow between phases in one generation.

Agent types

Six specialized roles, one evolving population

Each role contributes a different kind of variation. The orchestrator mixes them every generation so the population keeps genuine diversity — the thing vanilla best-of-N can't deliver.

Explore
Novel approaches — structurally different from existing solutions.
Exploit
Refine top solutions with targeted, local improvements.
Genetic
Crossover — combine strengths from two parent solutions.
Full
Full autonomy — read everything, try anything.
Research
Mathematical research — read papers, derive new approaches.
Experimentator
Builds shared helper utilities that every other agent can call.
Dashboard

Inspect any run live, straight from the filesystem

A Flask app reads directly from each run's directory — no database, no ingest job. It auto-detects every problem and attempt and renders six tabs: Overview, Pipeline, Architecture, Solutions, Knowledge, and Reports.

Architecture — run directory layout, knowledge hierarchy, idea lifecycle.

Solutions — every evaluated candidate, sortable, color-coded by validity.

Knowledge — three-layer hierarchy: State of Affairs → Clusters → Ideas / Patterns / Facts.

Reports — agent debrief notes and system feedback by generation.

Harnesses

Route any agent through any provider

Every agent session launches through a harness adapter. claude-code is the default; opencode is a second adapter that forwards to any provider OpenCode supports — Anthropic, OpenAI, Gemini, local models, or routed through a gateway like ModelGate. Both adapters expose the same contract, so timeout recovery and debrief work identically.

Override is per-role in a single YAML block — keep architects and evaluators on Claude for reasoning fidelity while routing explorers to a cheaper provider, or flip the default and keep only the architect on Claude. No code changes required.

Using it on your own problem

Drop a folder in problems/ and run the orchestrator

A new problem is a directory: description.md, constraints.md, an evaluate.py that scores candidates (results cached by content hash), a validate.py that enforces correctness, a metrics.yaml with fitness direction and target, plus helpers/ and initial_programs/. The shipped examples — gemm, permcodes, sidon — are the best starting templates.

python3 orchestrator.py . --problem sidon --new-attempt --single --dry-run
